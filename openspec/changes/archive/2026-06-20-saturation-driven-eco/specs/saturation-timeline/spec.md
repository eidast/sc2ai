## ADDED Requirements

### Requirement: Saturation timeline snapshots are captured during gameplay
The system SHALL append a saturation snapshot to the `saturation_timeline` list whenever the game time crosses a multiple of 60 seconds (same interval as `_build_army_snapshots`). Each snapshot SHALL include per-base saturation data and aggregate totals.

#### Scenario: Snapshot captured at 60s intervals
- **WHEN** game_time passes 60s, 120s, 180s, etc.
- **THEN** a saturation snapshot SHALL be appended with `time` equal to the game time and `bases` containing one entry per owned base

#### Scenario: Snapshot includes per-base mineral/gas breakdown
- **WHEN** a snapshot is captured for a base with 14 mineral workers and 4 gas workers
- **THEN** the base entry SHALL contain `mineral_workers: 14`, `gas_workers: 4`, `mineral_saturation`, `gas_saturation`, `status`, and `idle`

#### Scenario: Snapshot includes aggregate totals
- **WHEN** a snapshot is captured
- **THEN** the `totals` field SHALL include `workers`, `bases`, `oversaturated_bases`, `undersaturated_bases`, `idle_workers`, `avg_mineral_sat`, and `avg_gas_sat`

#### Scenario: Empty timeline for short matches
- **WHEN** a match ends before 60s of game time
- **THEN** `saturation_timeline` SHALL be an empty list

### Requirement: Saturation timeline is included in the match report
The system SHALL include the `saturation_timeline` field in the JSON report generated by `generate_report_json()`. The timeline SHALL be computed from features data by `_build_saturation_snapshots()`.

#### Scenario: Timeline present in report.json
- **WHEN** a match report is generated from a 15-minute game
- **THEN** `report.json` SHALL contain a `saturation_timeline` field with approximately 15 entries

#### Scenario: Timeline renders in HTML report
- **WHEN** the HTML report is generated
- **THEN** saturation data SHALL be displayed in a per-base cards section showing mineral and gas saturation bars with status labels

#### Scenario: Timeline data available for auto-tuner
- **WHEN** `tune_strategies.py` loads a `report.json`
- **THEN** it SHALL have access to `saturation_timeline` for computing temporal metrics like oversaturation duration, gas saturation timing, and idle worker minutes
